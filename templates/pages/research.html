Personal work on OS internals and low-level systems — the other side of how I think about reliability.

1 This is actually how I started programming. For 12–13 years I’ve wanted to understand how software works on the inside — and I grew up on PC games, so the two met naturally. I began with ChatEngine-era game exploits for fun, then kept going as modern protections got harder. Along the way I built this toolkit and several others (many unpublished) for my own experiments. Still a hobby first: curiosity about internals, not a product line.

Outside product work I spend time on Windows kernel and user-mode internals: drivers, memory access, and how privileged code talks to the rest of the system.1 That curiosity feeds how I reason about boundaries, failure modes, and performance in production platforms.

Windows Kernel Drivers

Kernel-mode driver experiments in C++ exploring user/kernel communication and IOCTL paths.

Injection & Overlays

Memory access, DLL loading, and real-time overlay rendering — mapping process space and drawing without owning the host UI.

Game client internals (archived)

Early research into a live game client: process attachment, DLL injection, memory layout, and overlay rendering — the same surfaces anti-cheat systems watch. Built as a Python tool with a custom loader; archived since 2023. Useful for understanding how untrusted code reaches another process’s address space.
